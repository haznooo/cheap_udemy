# cheap-udemy

## What this project is

A learning / CV project — a Udemy-style course platform API. **Not production-ready and never will be.** The goal is to practice and to have something to show on a CV. Prefer standered and common solutions over complex nich ones, even if it was a little limited.

## Current priority: usability pass, then frontend

The backend's architecture/hardening phase is considered **done** (auth, DI/interfaces, pagination, error contract, media lifecycle, refresh-token rotation — see "Key conventions" and the honest-state list below). Owner's stated goal now is **not** further refinement — it's making sure the API is actually **usable end-to-end** before switching primary focus to the `cheap-udemy-client` frontend ([[frontend-client-repo]] in memory). Concretely, that means finishing **CRUD completeness** on the core resources, not the leftover niche items (HtmlSanitizer wiring, Docker — those are explicitly deprioritized for now).

**CRUD completeness status as of 2026-07-11** (verified against controllers/services/schema, not assumed):

- **Sections** — complete. `POST section/add`, `GET {courseId}/sections`, `PUT section/{sectionId}` (rename/reorder), `DELETE section/{sectionId}` (hard delete, cascades to the section's lessons at the DB level). All owner/admin via `CheckCourseEditPermission`, allowed **regardless of enrollment** — same as lesson edits, only course-level hard delete is enrollment-gated.
- **Courses** — complete. `DELETE {courseId}` soft-deletes (`deleted_at`+`removal_reason`) a course, but **only if no one has ever enrolled** (any status, including dropped) — otherwise `409 Conflict`, instructor must unpublish instead. Publish/unpublish (`POST {courseId}/publish|unpublish`) covers the reversible state change; delete is for when a course has zero enrollment history and the instructor wants it fully gone.
- **Lessons** — complete (add/update/get/delete, owner/admin gated). No gap.
- **Enrollments** — complete (enroll/list-mine/list-by-user/list-by-course/mark-progress/read-progress). **Drop was removed entirely on 2026-07-17** (owner decision: "you do not drop the course") — no unenroll endpoint exists; legacy `dropped` DB rows still reactivate to `active` on re-enroll (`ReactivateDroppedEnrollmentAsync` kept for exactly that), and `HasAnyEnrollmentAsync` still counts them for the course-delete gate.

**Course lifecycle rules** (clarified/fixed 2026-07-11): a course with zero enrollment history can be hard-deleted (soft-delete columns) or unpublished, instructor's choice, no restrictions. A course with any enrollment history (even fully dropped) can only be unpublished, never deleted — unpublishing hides it from the catalog and blocks new enrollments, but **already-enrolled active/completed students keep full content access**, which required fixing `EnrollmentRepository.CanViewCourseContentAsync` (it used to gate on `course.status == "published"`, cutting off enrolled students the moment the course was unpublished — now it only checks `IsDeleted` + the caller's own enrollment status, see "Key conventions").

## Solution map

3 projects in `cheap_udemy.slnx`:

```
Api (learning_platform/Api.csproj)
  → Business (Application/Business.csproj)
    → DataAccess (data_access_layer/DataAccess.csproj)
```

- **Api** – ASP.NET Core Web API entrypoint (`Program.cs`), controllers, auth handlers, DI wiring
- **Business** – Services, request/response DTOs, the `MyResult<T>` result pattern (`Business.Common`)
- **DataAccess** – EF Core `AppDbContext`, entities, repositories, `clsPageResult.PageResult<T>` pagination, DB-level DTOs

> The `me/*` routes take no id and need no `UserOwnerOrAdmin` policy — identity is the token. The `api/admin` routes are gated by a single class-level `[Authorize(Roles="admin")]` (the codebase's first pure role gate; admin power elsewhere is still inline "owner-or-admin").

> **Stale-token guard on self-writes:** a JWT stays valid up to its 20-min expiry even after the account is deleted/banned (deletion nulls the password hash + sets `status='deleted'` but can't revoke an issued token; `/refresh` *is* closed — it fetches the user `active`-only and the trigger drops refresh tokens). So state-changing self-actions re-check the account is still active before mutating: `SetAvatar`/`RemoveAvatar`/`UpdatePassword`/`AddUserProfile`/`UpdateUserProfile` call `IsUserActiveAsync` first (→ 404 for a deleted/banned caller). `UpdatePassword` also enforces the same 5-20 length bounds as signup (BCrypt-72-byte rationale). This specifically stops the avatar upsert from **resurrecting a profile row** for a dead account. `DeleteUser` is intentionally **not** active-gated (it's shared by admin-delete, which must be able to delete a banned/suspended user). Not closed: cross-table writes in other controllers (courses/enrollments/reviews) still honor a stale token for ≤20 min — accepted, since the window is hard-capped and can't be extended.

## Key conventions (what's actually in use)

- **Result pattern**: Services return `MyResult<T>` (`Business.Common`) — internal-only, for passing errors between layers. Controllers never serialize it; they call `MapFailure(result)` on the shared base class.
- **Error contract (RFC 7807 ProblemDetails)**: **every** non-2xx response body is a ProblemDetails JSON (`{ type, title, status, detail, traceId }`) — the frontend reads `detail`. Produced in two places: (1) `ApiControllerBase.MapFailure` (`learning_platform/Controllers/ApiControllerBase.cs`) translates `MyResult` failures via `Problem(...)` — `NotFound` → 404, `BadRequest` → 400, `Conflict` → 409, and `Unauthorized` → **403** (service-level Unauthorized means "authenticated but not allowed", e.g. ownership/enrollment checks; only `AuthenticationController` passes 401 explicitly because failed login/refresh IS a credential problem — 401 tells the SPA "re-login", 403 tells it "not yours"); (2) middleware in `Program.cs` for everything that never reaches a controller: `AddProblemDetails()` + `UseExceptionHandler()` (unhandled exceptions → 500, no stack trace) + `UseStatusCodePages()` (empty-body 401/403 from the JWT middleware) + the rate limiter's `OnRejected` (429). Model-binding 400s are the framework's `ValidationProblemDetails` (same family; field errors under `errors` instead of `detail`). All controllers inherit `ApiControllerBase`, which also provides `CallerId`/`CallerRole` (from the JWT) and the `MissingIdentity()` 401 guard.
- **Validation**: done **manually** with inline `if` checks inside services (e.g. `if (request.UserId <= 0) return ...Failure(BadRequest, ...)`). FluentValidation was never wired up and has been **removed** from all `.csproj`s.
- **Mapping**: done **manually by hand** (e.g. `CoursesRepository.AddNewCourse` builds the DTO field-by-field). No source-generated mapper is actually in use.
- **Object creation — interfaces + constructor DI, all 7 vertical slices done**: every repository and service now follows the standard interface + constructor-DI pattern (prep for unit testing, no test project yet). Repo interfaces live in `data_access_layer/Interfaces/` (registered in `AddDataAccessDI`); service interfaces live in `Application/Interfaces/` (registered in `Application/DependencyInjection.cs` → `AddBusinessDI`, called from `AddApiDI`). All registrations are **Scoped** (matching `AppDbContext` — never Singleton, captive-dependency rule). Completed slices: **User** (`IUserAndProfileRepository`, `IUserService`, `IRefreshTokenService`), **Admin** (`IAdminService` — the admin **use-case** service, added 2026-07-17: `GetUser`/`DeleteUser`/`SetUserStatus` moved out of `UserService` so that one is self-service-only; **no new repo** — repos stay per-table, so `AdminService` sits on the existing `IUserAndProfileRepository` and writes its own audit rows via `IAdminActionService`, next to each mutation, instead of trusting the controller to remember), **AdminAction** (`IAdminActionRepository`, `IAdminActionService` — `AdminController` dropped `AppDbContext` entirely), **Auth** (`ILoginLogRepository`, `IRefreshTokenRepository`, `ILoginLogService`, `IAuthenticationService`; `RefreshTokenService`'s internals were migrated here too — ctor now `(IRefreshTokenRepository, IUserAndProfileRepository)`, no more `AppDbContext`), **Enrollment** (`IEnrollmentRepository`, `IEnrollmentService`), **Review** (`IReviewRepository`, `IReviewService` — needs `IEnrollmentRepository` too, for the enrollment check in `AddReview`), **Courses** (`ICoursesRepository` — gained a new `DoesCategoryExistAsync` method to replace `CourseService`'s old direct `context.Categories.AnyAsync` calls — and `ICourseService`, which needs `IEnrollmentRepository` too, for `GetCourseLessons`'s `CanViewCourseContentAsync` check), **Lessons** (`ILessonsRepository`, `ILessonService` — `LessonService` now injects `ICoursesRepository`, `IEnrollmentRepository`, and `ICourseService` directly instead of `new`ing a `CourseService` per call; `Program.cs`'s standalone `AddScoped<LessonService>()` was removed since `AddBusinessDI` now covers it via `ILessonService`). Statics stay off interfaces (e.g. `RefreshTokenService.HashRefreshToken` is still `public static`, called directly as `RefreshTokenService.HashRefreshToken(...)`). **Only remaining `new`-at-call-site holdout:** `CategoriesController` still queries `AppDbContext` directly (no service layer at all) — left as the lowest-priority loose end, give it a service only if full consistency is wanted. **Recipe for any future aggregate**, if one gets added: extract `IXxxRepository` into `DataAccess.Interfaces` (full public surface, signatures verbatim) + register in `AddDataAccessDI`; extract `IXxxService` into `Business.Interfaces` + register in `AddBusinessDI`; switch the service's per-method `new`s to injected ctor params (pull in sibling repo/service interfaces as needed — several slices above needed more than just "their own" repository); inject the service interface into its controller. See `docs/testing-di-pilot-plan.md` (fully superseded) for the abandoned earlier half-measure plan.
- **Pagination**: `clsPageResult.PageResult<T>` is a **single shared type** in `DataAccess.Common` (the old duplicate in `Business.Common` was deleted; services/controllers now `using (static) DataAccess.Common.clsPageResult` and just pass the repo's `PageResult` straight through — no more field-by-field copy across the layer boundary). **Why DataAccess and not Business:** it's the lowest layer (references nothing; everything above sees it), so repos can *return* the type directly and it flows up to the API with **zero mapping** — and it sits next to the `DataAccess.Dto` payloads it wraps (which already serialize straight to the client). Putting it in `Business` would forbid repos from returning it (DataAccess can't reference Business — backward dependency) and force the mapping copy back. `PageResult<T>` is a pure, dependency-free container (no EF/domain logic), so the lowest layer acts as the de-facto shared kernel. Trade-off accepted: the public API response shape is coupled to a DataAccess type; only worth revisiting if the API contract ever needs to stay stable while the DB layer changes. Applied consistently across **every** list endpoint now, including per-course sub-lists that used to return a bare `List<T>` — course lessons (`GET api/Courses/{courseId}/lessons`), enrollment progress (`GET api/Enrollments/progress/{courseId}`), and course reviews (`GET api/Courses/{courseId}/reviews/get`) all take `pageNumber`/`pageSize` query params (default 1/10) and return `PageResult<T>`, same as the catalog/enrollment-list endpoints. Owner's call: even though those sub-lists are bounded per-course, paging them anyway keeps every list endpoint uniform for API consumers (e.g. a future mobile client) and avoids unbounded response size if a course/review count grows large.
- **Error handling in repos**: repositories `try/catch (Exception)`, `Console.WriteLine` the error, and `return null`/`false`. So "DB failed" and "no rows" look identical to the caller. Repos still do **not** use `ILogger` (see Logging below for where `ILogger` _is_ used).
- **JSONB columns**: `LessonEntity.content_blocks`, `LessonEntity.lesson_metadata`, `CourseEntitiy.course_metadata`. Npgsql dynamic JSON serialization enabled in `data_access_layer/DependencyInjection.cs` (`NpgsqlConnection.GlobalTypeMapper.EnableDynamicJson()`).
- **Rich-text (text block) validation**: `TextBlockData.Content` (`data_access_layer/Entities/json/ContentBlockFormat.cs`) is still a plain `string`, but now holds **either** a legacy plain-text string **or** `JSON.stringify(doc)` of a constrained ProseMirror/TipTap doc (`{ "type": "doc", "content": [...] }`) written by the frontend editor. `LessonService.ValidateTextBlockContent` (called from `ValidateBlockData`, so both create & update paths are covered) validates it: content that doesn't start with `{`, or starts with `{` but doesn't parse as a JSON object whose root `type == "doc"`, is treated as **legacy plain text and accepted untouched** (hard requirement — pre-feature lessons must never break; a malformed doc that fails JSON parse is accepted as plain text rather than rejected, since it's indistinguishable from legacy prose that happens to start with `{`). Recognized docs are validated against an **allowlist subset**: block nodes `paragraph`/`codeBlock` under `doc`; `text` leaves only under those; marks limited to `bold`/`italic`/`textStyle`(hex color required)/`highlight`(hex color optional); `codeBlock.attrs.language` a short slug (`[A-Za-z0-9-+#.]`, ≤50 chars); text nodes non-empty. **Caps** (no request-size/DB-size limit exists otherwise): depth ≤20, ≤5000 nodes, ≤50,000 total leaf-text chars. Violations return one `BadRequest` message via the existing `ValidateBlockData` error list — same response shape as image/video/quiz block validation, no new failure modes.
- **Auth**: JWT bearer with a custom `UserOwnerOrAdmin` policy + handler (checks user ID match or admin role). Issuer `CheapUdemyApi`, audience `CheapUdemyApiUsers`, expiry 20 min. Refresh tokens rotated via `RefreshTokenService`, stored in `user_refresh_tokens` as a **deterministic SHA-256 hash** (`RefreshTokenService.HashRefreshToken`; high-entropy random tokens don't need a salt — **only passwords use BCrypt**). Self-actions take the user id from the **access token** (`me/*` routes), not the URL — so there's no per-request ownership check to get wrong (anti-IDOR). Cross-user/admin access uses `[Authorize(Roles="admin")]`. The `UserOwnerOrAdmin` policy + handler were **deleted** 2026-07-17 (registered but had zero call sites). **Emails are normalized to lowercase** at signup + login (same as usernames; existing DB rows confirmed already-lowercase). Signup/login now **fail loudly (500)** if the refresh-token mint fails instead of returning 200 with a null RefreshToken. See **"Refresh-token rotation"** below for the full `/refresh` flow (reuse detection + absolute expiry).
- **Rate limiting** (in use): configured in `Program.cs` via `AddRateLimiter`, using **named policies only — no `GlobalLimiter`**. Two fixed-window policies, both partitioned by `RemoteIpAddress`, `QueueLimit = 0`, rejection status `429`: **`"standard"`** (**67 requests/min per IP**) and **`"auth"`** (**10 requests/min per IP**, shared across `login`/`signUp`/`refresh` — one counter for all three, not one each). `"standard"` is applied via a **class-level** `[EnableRateLimiting("standard")]` on `ApiControllerBase` (`learning_platform/Controllers/ApiControllerBase.cs`), so every controller gets it by default through inheritance; `login`/`signUp`/`refresh` override it with a **method-level** `[EnableRateLimiting("auth")]` (method-level metadata wins over class-level, so each endpoint resolves to exactly one policy — never both). `logout` has no override, so it stays on `"standard"`. **Why not `GlobalLimiter` + `[DisableRateLimiting]`:** that was the first attempt and is a real bug trap — `GlobalLimiter` always stacks (AND's) with any endpoint policy, and empirically `[DisableRateLimiting]` disables the global limiter too when combined with `[EnableRateLimiting]` on the same endpoint, silently leaving the endpoint with **zero** rate limiting (verified: 75 requests, no `429`s). The two-named-policies-with-inheritance-override approach is the only way in ASP.NET Core to get truly independent per-endpoint budgets. Both policies are plain attempt counters (count successes and failures alike, not failure-only lockout — a deliberate, simpler tradeoff). `app.UseRateLimiter()` runs _before_ auth; the limiter's `OnRejected` writes a friendly ProblemDetails body on 429 (no exact limits exposed). Relies on `UseForwardedHeaders` for the client IP behind a proxy (trusted-proxy restriction is still a TODO, so `X-Forwarded-For` is currently spoofable).
- **Logging (`ILogger`)**: used in **controllers** for security events only — failed logins (`LogWarning`, IP only, never credentials) and failed refresh attempts in `AuthenticationController`, and successful admin deletes (`LogInformation`) in `AdminController`. `ILogger<T>` is injected via the controllers' primary constructors. Repositories still use `Console.WriteLine` (not migrated).
- **Login logging** (DB): `LoginLogService` → `LoginLogRepository` write a row to `login_logs` (`success`/`failed`) on signup and login attempts.
- **Admin-action auditing** (DB): `AdminActionService` → `AdminActionRepository` write an immutable row to `admin_actions` (mirrors the login-log `new`d pattern). Uploads return the stored file name (a GUID + extension, always server-generated — **no request DTO anywhere accepts a client-supplied file name**). Bucket names are constants in `MediaBuckets`, used only inside the service. Public buckets are read back via the permanent public object URL `{Supabase:Url}/storage/v1/object/public/{bucket}/{fileName}`. The upload sets `ContentType = file.ContentType` explicitly — the supabase-csharp default is `text/plain`, which MIME-restricted buckets reject. There is **no generic upload endpoint**. Three ownership-scoped upload paths: `PUT api/Courses/{courseId}/thumbnail` (JPG/PNG ≤5 MB) → `courses.thumbnail_url`, `POST api/user/me/avatar` (JPG/PNG ≤3 MB, self) → `users_profile.image_url`, and `POST api/Courses/{courseId}/media` (JPG/PNG/MP4/MOV ≤5 MB) → returns the file name for embedding in lesson `content_blocks` (owner/admin, permission checked **before** upload). **Profile add/update never touch `image_url`**; the avatar changes only via the avatar endpoint. **API read-back note:** responses return the raw stored file name (`ImageUrl`, `thumbnail_url`); the frontend builds the public URL itself.

Cleanup paths: avatar replace + avatar remove (`DELETE me/avatar`) + account delete (self & admin; the file name is captured **before** `DeleteUserAsync_Anonymize`, because the DB trigger deletes the profile row), thumbnail replace, lesson update (old vs new `content_blocks` file-name diff), and lesson delete. Lesson media is only deleted if **no other lesson still references it** — `LessonsRepository.IsMediaReferencedByOtherLessonsAsync` does a raw-SQL `content_blocks::text LIKE '%name%'` check (GUID names ⇒ no false positives; on query error it reports "referenced" so the file is kept). Only supabase-hosted block media counts (`ImageBlockData.Url`, `VideoBlockData` with `Provider == "supabase"`); external URLs/ids are never deleted, and `DeleteFromBucketAsync` refuses names containing path separators. **Accepted leaks (deliberate, don't “fix” without asking):** files uploaded via `POST {courseId}/media` but never saved into a lesson; old thumbnails uploaded before the `course-thumbnail` bucket existed (they sit in `course-media`); partially-failed cleanup; lesson media belonging to lessons removed via `DELETE api/Courses/section/{sectionId}`'s DB cascade (that path never runs the per-lesson media cleanup `DeleteLessonAsync` does). A background orphan-sweep job would cover these but is out of scope for this project.

## Refresh-token rotation (implemented)

The `/refresh` flow (`api/User/refresh` → `RefreshTokenService.RefreshAccessToken`) hardens a *stolen* refresh token so it's actually containable, not just time-limited. `/refresh` and `/logout` both take a `RefreshTokenRequest` of just **`{ RefreshToken }`** in the body; the (possibly expired) access token is sent in the standard **`Authorization: Bearer <token>`** header, exactly like every other endpoint (no client-supplied user id, no token in the body). `/refresh` returns a slim **`RefreshResponse` = `{ AccessToken, RefreshToken, RefreshTokenExpiresAt }`** — **tokens only, no user/profile info** (refresh is a token exchange; the client already knows who it is, so echoing identity back is needless exposure).

### User id comes from the access token, never the body

The controller reads the bearer token via `AuthenticationController.GetBearerToken` (raw `Authorization` header — the auth middleware can't be used since it rejects expired tokens, and refresh deliberately accepts them), then `GetUserIdFromExpiredToken` validates its **signature/issuer/audience** but sets `ValidateLifetime = false`, so an *expired* access token still works as the trustworthy source of the user id (`NameIdentifier`). A tampered/garbage token → `null` → `401`. The header token scopes the refresh-token lookup.

### Lookup & state branching

Tokens are stored as deterministic SHA-256 hashes, so `RefreshTokenRepository.GetRefreshTokenByHashAsync(userId, tokenHash)` does a **direct indexed lookup across ALL states** (used/revoked/expired included — a valid-only filter would hide a replayed token). `RefreshAccessToken` then branches on the found row:

| State | Outcome |
| ----- | ------- |
| not found | `401` invalid, no breach |
| `replaced_by_id != null` (superseded, replayed) | **BREACH** → revoke whole chain → `401 "reuse detected"` |
| `is_used` / `revoked_at` set (e.g. logged out), or expired | `401`, plain re-login, no breach |
| valid | rotate |

**The breach trigger is `replaced_by_id != null`, NOT `is_used`** — deliberate: logout sets `is_used = true` but leaves `replaced_by_id == null`, so a logged-out token falls through to a plain `401` instead of false-flagging as theft. Only a token actually rotated to a child can trip the breach.

### Reuse detection (the anti-theft measure)

`RevokeBreachedChainAsync` walks `replaced_by_id` **forward** from the replayed token, setting `chain_breached + is_used + revoked_at` on every link — including the thief's currently-valid token. Scoped to the chain via forward links, **never by `user_id`**, so the user's other devices (separate chains) stay logged in. Forward-only (each child has a higher id) ⇒ no cycles. Net: a thief forces the user to re-login exactly once per theft; the new chain is one the thief doesn't hold.

### Absolute expiration (no sliding reset)

`AddNewRefreshTokenFirstTime(userId, deviceInfo, ipAddress, DateTime? expiresAt = null)` is the shared minter. Login/signup pass no expiry → fresh `AddDays(7)`. Rotation passes `currentToken.expires_at` → the child **inherits the parent's deadline**, so the original-login deadline propagates down the chain and a session (legit or stolen) dies ≤7 days after the original login, full stop.

### Bugs fixed in the same method

- **Bug 1:** the user is now fetched + validated **before** any mint/revoke (a deleted/banned user fails with a clean `401` instead of burning the old token + orphaning a new one). `UserAndProfileRepository.GetUserByIdAsync` got a null guard (the `status == "active"` filter returns no row → previously NRE → 500).
- **Bug 2:** the revoke (`UpdateRefreshTokenAsync`) result is checked, so a silent failure can't leave two valid tokens.

### Explicitly OUT OF SCOPE (owner's decision — this is a CV project)

- **Benign double-refresh grace window.** Strict reuse detection can false-positive when an honest client fires `/refresh` twice near-simultaneously (two tabs, retries) and the second call sees the now-used token. **Deliberately NOT handled.** Do not add it.
- **No transaction** around the mint+revoke pair or the breach-revoke loop. Each `UpdateRefreshTokenAsync` saves independently, so a mid-loop failure can leave a chain *partially* revoked (risk = under-revoking, e.g. the thief's link survives) — accepted as-is.

## No tests / how to verify manually

There is no test project. All verification is manual via the HTTP API (see `learning_platform.http` or the Scalar UI at `/scalar/v1`). Useful checks for the recently-added features:

- **Rate limiting**: hammer any non-auth endpoint >67×/min from the same IP → expect `429` + a ProblemDetails body ("Too many requests. Please try again later."). Hammer `login`/`signUp`/`refresh` (mix of all three) >10×/min from the same IP → `429` much sooner, independent of the 67/min global bucket. Both counters are per-IP, fixed 1-min window.
- **Error contract**: any failing request returns `application/problem+json`; quick checks — no token on `GET api/user/me` → 401 ProblemDetails (from `UseStatusCodePages`), wrong password on login → 401 with `detail: "invalid credentials"`, `GET api/Courses/999999` → 404 with `detail: "Course not found."`, malformed JSON body → 400 `ValidationProblemDetails`.
- **Failed-login logging**: POST `api/User/login` with a wrong password → `LogWarning` in console (IP only) **and** a `failed` row in `login_logs`.
- **403 logging**: call an `api/user/...` endpoint with a JWT for a _different_ user (non-admin) → `403` + `LogWarning`.
- **Admin-action auditing**: log in as an `admin`, delete _another_ user via `DELETE api/admin/users/{userId}` (no password body) → a `delete` row appears in `admin_actions` (immutable; FK/trigger requires the caller's `admin_id` to actually have role `admin` in the DB, or the audit insert is silently swallowed by the repo's try/catch). Self-deletion does **not** create an audit row.
- **Ban/suspend/unban**: as admin, `POST api/admin/users/{userId}/ban` on a student → `204`, a `ban` row in `admin_actions` with `{ status }` old/new JSONB, all the target's `user_refresh_tokens` revoked, and the target's login returns the generic 401. Unban a *suspended* user → the audit row is `unsuspend` (old status decides). Edge cases: self → 400, another admin → 400, deleted user → 404, repeat of the same state → 409. Read it all back via `GET api/admin/actions` (paged, newest first). All verified end-to-end 2026-07-12.
