# cheap-udemy

## What this project is

A learning / CV project ŌĆö a Udemy-style course platform API. **Not production-ready and never will be.** The goal is to practice and to have something to show on a CV. Prefer standered and common solutions over complex nich ones, even if it was a little limited.

## Current priority: usability pass, then frontend

The backend's architecture/hardening phase is considered **done** (auth, DI/interfaces, pagination, error contract, media lifecycle, refresh-token rotation ŌĆö see "Key conventions" and the honest-state list below). Owner's stated goal now is **not** further refinement ŌĆö it's making sure the API is actually **usable end-to-end** before switching primary focus to the `cheap-udemy-client` frontend ([[frontend-client-repo]] in memory). Concretely, that means finishing **CRUD completeness** on the core resources, not the leftover niche items (HtmlSanitizer wiring, Docker ŌĆö those are explicitly deprioritized for now).

**CRUD completeness status as of 2026-07-11** (verified against controllers/services/schema, not assumed):

- **Sections** ŌĆö complete. `POST section/add`, `GET {courseId}/sections`, `PUT section/{sectionId}` (rename/reorder), `DELETE section/{sectionId}` (hard delete, cascades to the section's lessons at the DB level). All owner/admin via `CheckCourseEditPermission`, allowed **regardless of enrollment** ŌĆö same as lesson edits, only course-level hard delete is enrollment-gated.
- **Courses** ŌĆö complete. `DELETE {courseId}` soft-deletes (`deleted_at`+`removal_reason`) a course, but **only if no one has ever enrolled** (any status, including dropped) ŌĆö otherwise `409 Conflict`, instructor must unpublish instead. Publish/unpublish (`POST {courseId}/publish|unpublish`) covers the reversible state change; delete is for when a course has zero enrollment history and the instructor wants it fully gone.
- **Lessons** ŌĆö complete (add/update/get/delete, owner/admin gated). No gap.
- **Enrollments** ŌĆö complete (enroll/list-mine/list-by-user/list-by-course/mark-progress/read-progress). The **course roster** (`GET api/Enrollments/course/{courseId}`, owner/admin) returns `PageResult<CourseEnrollmentDto>` (added 2026-07-22) ŌĆö a distinct DTO carrying the **student's identity** (`UserId`/`Username`/`DisplayName`/`ImageUrl`, the last two null when the student has no profile row, LEFT-joined via `e.user.UserProfile`) plus enrollment fields (dates/status/progress), ordered newest-first; the shared `EnrollmentDto` (used by the *my-enrollments* reads) was deliberately left alone since there the student is the caller. **Drop was removed entirely on 2026-07-17** (owner decision: "you do not drop the course") ŌĆö no unenroll endpoint exists; legacy `dropped` DB rows still reactivate to `active` on re-enroll (`ReactivateDroppedEnrollmentAsync` kept for exactly that), and `HasAnyEnrollmentAsync` still counts them for the course-delete gate.

**Course lifecycle rules** (clarified/fixed 2026-07-11): a course with zero enrollment history can be hard-deleted (soft-delete columns) or unpublished, instructor's choice, no restrictions. A course with any enrollment history (even fully dropped) can only be unpublished, never deleted ŌĆö unpublishing hides it from the catalog and blocks new enrollments, but **already-enrolled active/completed students keep full content access**, which required fixing `EnrollmentRepository.CanViewCourseContentAsync` (it used to gate on `course.status == "published"`, cutting off enrolled students the moment the course was unpublished ŌĆö now it only checks `IsDeleted` + the caller's own enrollment status, see "Key conventions").

**Course suspension (admin moderation, added 2026-07-19):** `courses.status` gained a fourth value `'suspended'` (CHECK updated in schema file + live DB). `POST api/admin/courses/{courseId}/suspend|unsuspend` (admin-only, audited as `suspend`/`unsuspend` on `target_table: "courses"`, via `AdminService.SetCourseSuspension` ŌĆö `AdminService` now also injects `ICoursesRepository`). **Unlike unpublish, suspension blocks EVERYONE including enrolled students** (`CanViewCourseContentAsync` returns false for suspended, after the owner/admin bypass ŌĆö so the owner/admin can still view it; progress read/write also 404 on suspended, and mark-progress now fetches course info to check). Only a **published** course can be suspended (draft/retired is invisible anyway ŌåÆ 409), unsuspend always restores exactly `published` (no previous-status column needed). The instructor **cannot** lift it: `CourseService.PublishCourse` rejects a suspended course with 409 "suspended by an administrator". Deleted course ŌåÆ 404; already/not suspended ŌåÆ 409. Enrolling in a suspended course was already blocked (`Status != "published"` ŌåÆ 404 in `EnrollStudent`), same for new reviews (`AddReview` requires published). **Reviews are suspension-gated too** (security-audit finding, fixed same day): `GetCourseReviews`/`GetMyReview`/`UpdateReview` all 404 on a suspended (or deleted) course ŌĆö without this the reviews endpoint leaked that a suspended course exists and let the author keep editing; `GetCourseReviews` now fetches course info via `GetCourseEnrollmentInfoAsync` (the unfiltered `ReviewRepository.GetCourseInstructorIdAsync` it used before was left with zero call sites and deleted). `DeleteReview` stays ungated (removing a review isn't exposure). Known accepted escape: an instructor can still soft-delete their own suspended course if it has zero enrollment history ŌĆö deletion hides harder than suspension, and the audit row survives, so the moderation intent holds.

**Simulated payments (added 2026-07-19):** paid courses are now purchasable ŌĆö `POST api/Payments/checkout { CourseId }` (authenticated; caller from the JWT) records a `completed` payment (provider `simulated`, `provider_reference` = `SIM-<guid>`, amount = course price, USD) and enrolls the caller in one step; `GET api/Payments/me` is the caller's paged payment history (`PageResult<PaymentDto>`, newest first). New vertical slice via the standard recipe: `IPaymentRepository`/`PaymentRepository`, `IPaymentService`/`PaymentService` (injects `IEnrollmentRepository` for the course/enrollment checks), `PaymentController`. Checkout guards: course must be published/non-deleted (draft/retired/suspended/deleted ŌåÆ 404, same visibility rule as `EnrollStudent`), not the caller's own course ŌåÆ 400, not free ŌåÆ 400 (free courses use the enroll endpoint; `EnrollStudent`'s paid-course 400 message now points at checkout), already enrolled ŌåÆ 409 (a legacy `dropped` enrollment is reactivated instead, like the free path). Write order follows the decided plan ŌĆö payment row first, enrollment second, then `payments.enrollment_id` linked back ŌĆö with **no transaction** around the steps (same accepted trade-off as the refresh-token mint/revoke pair); each step fails loudly (500 with a service-authored `detail`) rather than succeeding silently. No refund/pending states are exposed ŌĆö checkout is the entire simulated flow, and there is no real gateway or webhook. DB note: `payments.payment_date` was the only `timestamp`-without-tz column (every other table uses `timestamptz`, and the EF config already said `timestamp with time zone`) ŌĆö ALTERed to `timestamptz` (schema file + live DB, applied 2026-07-19 while the table was empty). Verified end-to-end 2026-07-19 (checkout/duplicate/own-course/free-course/anonymous/missing-course + history + resulting enrollment).

**Lesson publish workflow (added 2026-07-19):** the previously-unmapped `lessons.status` column is now used ŌĆö `'draft'` (never published, the status every new lesson is created with) ŌåÆ `'published'` (visible to students) ŌåÆ `'hidden'` (unpublished after being live; re-publish goes back to `'published'`). `POST api/Lessons/{id}/publish|unpublish` (owner/admin via lessonŌåÆsectionŌåÆcourse `CheckCourseEditPermission`; publish on already-published ŌåÆ 409, unpublish on non-published ŌåÆ 409; returns the `LessonDto`, which now carries `Status`). **Draft/hidden lessons are invisible to enrolled students**: `GET api/Lessons/get/{id}` 404s them for non-owner/admin, `GET api/Courses/{courseId}/lessons` filters to published for students (owner/admin see all ŌĆö `CoursesRepository.GetCourseLessons` gained an `includeUnpublished` param), progress read (`GetUserCourseProgressAsync`) lists published lessons only, and mark-progress 404s a non-published lesson (`IEnrollmentRepository.GetLessonStatusAsync`). The `trg_sync_progress` DB trigger now computes `progress_percentage` over **published lessons only** (schema file + live `CREATE OR REPLACE` both applied) ŌĆö otherwise a course with any draft lesson could never reach 100%. Accepted staleness: percentages don't recompute when a lesson's status changes, only on the next progress write; `course_metadata.lessons_count` still counts all lessons regardless of status. Existing DB rows were deliberately **left as `'draft'`** (owner's call ŌĆö the 2 test lessons just need publishing via the new endpoint). `UpdateLessonRequest` deliberately has no `Status` field ŌĆö status changes only via the publish/unpublish endpoints, same as courses.

> The `me/*` routes take no id and need no `UserOwnerOrAdmin` policy ŌĆö identity is the token. The `api/admin` routes are gated by a single class-level `[Authorize(Roles="admin")]` (the codebase's first pure role gate; admin power elsewhere is still inline "owner-or-admin").

> **Stale-token guard on self-writes:** a JWT stays valid up to its 20-min expiry even after the account is deleted/banned (deletion nulls the password hash + sets `status='deleted'` but can't revoke an issued token; `/refresh` *is* closed ŌĆö it fetches the user `active`-only and the trigger drops refresh tokens). So state-changing self-actions re-check the account is still active before mutating: `SetAvatar`/`RemoveAvatar`/`UpdatePassword`/`AddUserProfile`/`UpdateUserProfile` call `IsUserActiveAsync` first (ŌåÆ 404 for a deleted/banned caller). `UpdatePassword` also enforces the same 5-20 length bounds as signup (BCrypt-72-byte rationale). This specifically stops the avatar upsert from **resurrecting a profile row** for a dead account. `DeleteUser` is intentionally **not** active-gated (it's shared by admin-delete, which must be able to delete a banned/suspended user). Not closed: cross-table writes in other controllers (courses/enrollments/reviews) still honor a stale token for Ōēż20 min ŌĆö accepted, since the window is hard-capped and can't be extended.

## Key conventions (what's actually in use)

- **Result pattern**: Services return `MyResult<T>` (`Business.Common`) ŌĆö internal-only, for passing errors between layers. Controllers never serialize it; they call `MapFailure(result)` on the shared base class.
- **Error contract (RFC 7807 ProblemDetails)**: **every** non-2xx response body is a ProblemDetails JSON (`{ type, title, status, detail, traceId }`) ŌĆö the frontend reads `detail`. Produced in two places: (1) `ApiControllerBase.MapFailure` (`Api/Controllers/ApiControllerBase.cs`) translates `MyResult` failures via `Problem(...)` ŌĆö `NotFound` ŌåÆ 404, `BadRequest` ŌåÆ 400, `Conflict` ŌåÆ 409, and `Unauthorized` ŌåÆ **403** (service-level Unauthorized means "authenticated but not allowed", e.g. ownership/enrollment checks; only `AuthenticationController` passes 401 explicitly because failed login/refresh IS a credential problem ŌĆö 401 tells the SPA "re-login", 403 tells it "not yours"; `ErrorType.Failure` ŌåÆ **500 with the service-authored `detail`** ŌĆö safe to surface since repos swallow exceptions, so raw exception text never reaches `MapFailure`; e.g. the refresh-token-mint failure message on signup/login. Relatedly, `AuthenticationController.login` only `LogWarning`s a "failed login attempt" on `ErrorType.Unauthorized` ŌĆö a server-side 500 doesn't pollute the failed-login signal); (2) middleware in `Program.cs` for everything that never reaches a controller: `AddProblemDetails()` + `UseExceptionHandler()` (unhandled exceptions ŌåÆ 500, no stack trace) + `UseStatusCodePages()` (empty-body 401/403 from the JWT middleware) + the rate limiter's `OnRejected` (429). Model-binding 400s are the framework's `ValidationProblemDetails` (same family; field errors under `errors` instead of `detail`). All controllers inherit `ApiControllerBase`, which also provides `CallerId`/`CallerRole` (from the JWT) and the `MissingIdentity()` 401 guard.
- **Validation**: done **manually** with inline `if` checks inside services (e.g. `if (request.UserId <= 0) return ...Failure(BadRequest, ...)`). FluentValidation was never wired up and has been **removed** from all `.csproj`s.
- **Mapping**: done **manually by hand** (e.g. `CoursesRepository.AddNewCourse` builds the DTO field-by-field). No source-generated mapper is actually in use.
- **Object creation ŌĆö interfaces + constructor DI, all 7 vertical slices done**: every repository and service now follows the standard interface + constructor-DI pattern (prep for unit testing, no test project yet). Repo interfaces live in `DataAccess/Interfaces/` (registered in `AddDataAccessDI`); service interfaces live in `Business/Interfaces/` (registered in `Business/DependencyInjection.cs` ŌåÆ `AddBusinessDI`, called from `AddApiDI`). All registrations are **Scoped** (matching `AppDbContext` ŌĆö never Singleton, captive-dependency rule). ## Refresh-token rotation (implemented)

The `/refresh` flow (`api/User/refresh` ŌåÆ `RefreshTokenService.RefreshAccessToken`) hardens a *stolen* refresh token so it's actually containable, not just time-limited. `/refresh` and `/logout` both take a `RefreshTokenRequest` of just **`{ RefreshToken }`** in the body; the (possibly expired) access token is sent in the standard **`Authorization: Bearer <token>`** header, exactly like every other endpoint (no client-supplied user id, no token in the body). `/refresh` returns a slim **`RefreshResponse` = `{ AccessToken, RefreshToken, RefreshTokenExpiresAt }`** ŌĆö **tokens only, no user/profile info** (refresh is a token exchange; the client already knows who it is, so echoing identity back is needless exposure).

### User id comes from the access token, never the body

The controller reads the bearer token via `AuthenticationController.GetBearerToken` (raw `Authorization` header ŌĆö the auth middleware can't be used since it rejects expired tokens, and refresh deliberately accepts them), then `GetUserIdFromExpiredToken` validates its **signature/issuer/audience** but sets `ValidateLifetime = false`, so an *expired* access token still works as the trustworthy source of the user id (`NameIdentifier`). A tampered/garbage token ŌåÆ `null` ŌåÆ `401`. The header token scopes the refresh-token lookup.

### Lookup & state branching

Tokens are stored as deterministic SHA-256 hashes, so `RefreshTokenRepository.GetRefreshTokenByHashAsync(userId, tokenHash)` does a **direct indexed lookup across ALL states** (used/revoked/expired included ŌĆö a valid-only filter would hide a replayed token). `RefreshAccessToken` then branches on the found row:

| State | Outcome |
| ----- | ------- |
| not found | `401` invalid, no breach |
| `replaced_by_id != null` (superseded, replayed) | **BREACH** ŌåÆ revoke whole chain ŌåÆ `401 "reuse detected"` |
| `is_used` / `revoked_at` set (e.g. logged out), or expired | `401`, plain re-login, no breach |
| valid | rotate |

**The breach trigger is `replaced_by_id != null`, NOT `is_used`** ŌĆö deliberate: logout sets `is_used = true` but leaves `replaced_by_id == null`, so a logged-out token falls through to a plain `401` instead of false-flagging as theft. Only a token actually rotated to a child can trip the breach.

### Reuse detection (the anti-theft measure)

`RevokeBreachedChainAsync` walks `replaced_by_id` **forward** from the replayed token, setting `chain_breached + is_used + revoked_at` on every link ŌĆö including the thief's currently-valid token. Scoped to the chain via forward links, **never by `user_id`**, so the user's other devices (separate chains) stay logged in. Forward-only (each child has a higher id) ŌćÆ no cycles. Net: a thief forces the user to re-login exactly once per theft; the new chain is one the thief doesn't hold.

### Absolute expiration (no sliding reset)

`AddNewRefreshTokenFirstTime(userId, deviceInfo, ipAddress, DateTime? expiresAt = null)` is the shared minter. Login/signup pass no expiry ŌåÆ fresh `AddDays(7)`. Rotation passes `currentToken.expires_at` ŌåÆ the child **inherits the parent's deadline**, so the original-login deadline propagates down the chain and a session (legit or stolen) dies Ōēż7 days after the original login, full stop.

### Bugs fixed in the same method

- **Bug 1:** the user is now fetched + validated **before** any mint/revoke (a deleted/banned user fails with a clean `401` instead of burning the old token + orphaning a new one). `UserAndProfileRepository.GetUserByIdAsync` got a null guard (the `status == "active"` filter returns no row ŌåÆ previously NRE ŌåÆ 500).
- **Bug 2:** the revoke (`UpdateRefreshTokenAsync`) result is checked, so a silent failure can't leave two valid tokens.

### Explicitly OUT OF SCOPE (owner's decision ŌĆö this is a CV project)

- **Benign double-refresh grace window.** Strict reuse detection can false-positive when an honest client fires `/refresh` twice near-simultaneously (two tabs, retries) and the second call sees the now-used token. **Deliberately NOT handled.** Do not add it.
- **No transaction** around the mint+revoke pair or the breach-revoke loop. Each `UpdateRefreshTokenAsync` saves independently, so a mid-loop failure can leave a chain *partially* revoked (risk = under-revoking, e.g. the thief's link survives) ŌĆö accepted as-is.

## Database setup

- `db/01-create-schema.sql` ŌĆō full schema with tables, constraints, indexes, triggers (the **source of truth for local dev**; starts with `DROP SCHEMA public CASCADE`, so it wipes the DB). No RLS ŌĆö a local Postgres isn't exposed to any untrusted role.
- `db/01-create-schema.supabase.sql` ŌĆō **Supabase-hosted variant** (the live demo DB runs on Supabase ŌĆö see README "Deployment"). Differs from the local file in three ways: (1) drops the **tables** instead of the `public` schema (dropping the schema resets Supabase's managed role grants); (2) `SET search_path TO public, extensions` so the trigram index resolves wherever `pg_trgm` lives; (3) **enables deny-all RLS on all 13 tables** at the end, to hide the data from Supabase's public auto-generated PostgREST API (`anon`/`authenticated` roles) ŌĆö the app connects as the `postgres` table owner, which **bypasses RLS**, so app behaviour is unchanged. Keep it in sync by hand when the schema changes; regenerate from `01-create-schema.sql` + these three deltas.
- `db/02-seed.sql` ŌĆō seed data (categories + one admin user; Supabase-compatible as-is).
- EF Core uses `Npgsql`. **No EF migrations** ŌĆö schema is managed via the raw SQL scripts above. Schema changes are applied to **both** the SQL file(s) **and** the live DB by hand (the long-standing "schema file + live-DB, both applied" habit); on the live Supabase DB, apply incremental `ALTER`/`CREATE` deltas via the SQL Editor ŌĆö **never** re-run `01-create-schema.supabase.sql` against data you want to keep, and remember to `ENABLE ROW LEVEL SECURITY` on any **new** table.
- **Naming**: no naming-convention package ŌĆö entity **property names are written literally in snake_case** (e.g. `user_id`, `target_table`) so they match the DB columns 1:1 (`EFCore.NamingConventions` was referenced early on but never configured, and was removed as dead weight). **Gotcha:** a typo'd property name therefore maps to a non-existent column and fails at runtime (this happened with `admin_actions.trget_table` ŌåÆ fixed to `target_table` + explicit `HasColumnName`). When adding columns, match the SQL column name exactly or add `.HasColumnName(...)`.
